git 简单使用介绍

简介

git 是一个分布式的版本管理系统，最初由linux的开发者Linus Benedict Torvalds创建。

简单配置

git config --global user.name "your name"
git config --global user.email "your_email@youremail.com"

让 Git 知道你是谁

基本操作

  • 初始化仓库
git init

进入要进行版本控制的文件夹，打开git bash,输入以上命令。这样你就在本地创建了一个仓库。

现在让我们向本地仓库添加一个文件——readme.txt,并写入 git is a version controll.

现在我们把他添加至“暂存区”

  • 保存改动至“暂存区”
git add <filename> //git add readme.txt
git add * // 当你修改了多个文件时你可以使用系命令一次性添加

如果不懂暂存区是什么也没有关系。
保存至“暂存区”并不意味着提交，因此你需要使用下面的命令提交你的修改。

  • 提交修改
git commit -m "commit message"
git commit -m "添加了readme.txt" //-m 参数后面写改动的内容

这样你就完成了本地仓库的一次提交

  • 查看信息
git status

你会看到如下输出

On branch master //当前master分支，也是git默认创建出来的分支
nothing to commit, working tree clean
  • 创建新的分支
git branch 
git branch testing

或者

git checkout -b "branch name" //创建分支并切换到新创建的分支

创建testing分支

  • 查看分支
git branch -a

* master
  testing
//* 表示当前分支
  • 切换分支
git checkout "branch name"
git checkout "testing"

我们在testing分支继续修改readme.txt文件,添加“I am Iron man”
使用 add&commit 命令 保存和提交

  • 查看提交信息
git log

这个命令可以使你看到历史修改信息

现在我们再次切换回 master 分支

git checkout master

你发现了什么？——readme.txt 里还是最初添加的内容
我们可以得出这样的结论，不同分支间的内热是互不影响的

  • 合并分支
git checkout master
git merge testing // 把testing 分支合并到 master 分支

将在 testing 分支的修改合并到master 分支，这样两个分支的版本就一致了
Fast-forward 表示没有冲突

  • 删除分支
git branch
git branch -d "testing"
  • 合并冲突
git merge

只保留你想要留下的改动
使用 git add 表明你已经解决了冲突
使用 git commit 提交

  • 删除分支
git branch -d
git branch -d "testing"

版本回退

日常操作中我们都可能会有犯错误的时候，这时该怎么办?

别紧张，git早就猜到，所以可以帮助你撤销操作。

我们修改readme.txt，添加如下内容

“fuck life”

当你在修改完之后心想：“奥，我的上帝~，我不该这么消极”，于是你想撤销修改

git checkout filename
git checkout .

此方法适用于修改而未add，实测可以不加--也能修改

当你不小心已执行add，也没有关系，我们使用git status查看状态，发现一条提示

git reset HEAD <file>

我们执行git reset HEAD readme.txt,效果如下所示

执行git status我们发现，readme.txt文件回到了modified状态，wow aswsome!

此时你依然使用git checkout readme.txt去撤销工作区中文件内容的修改

提示

当add了多个文件，如果想撤销全部使用git reset HEAD .

此时由于你粗心，可能在commit操作完成后你才发现大事不妙，这时该怎么办呢？

不要慌，我们先来看一下情况，使用git log查看历史commit操作

我们发现每个commit后有一串黄色的看不懂的字符串，它就是commit id，你也不需要看懂，只是记录标记提交而已

接下来我们就通过commit id回到过去

git reset --hard <commit id>

你只需使用commit id的前几位即可

版本号没必要写全，前几位就可以了，Git会自动去找。当然也不能只写前一两位，因为Git可能会找到多个版本号，就无法确定是哪一个了。 ——廖雪峰老师的官网

我们再查看一下readme.txt里面的内容，你会发现回到了原始状态

如果只是回退到上一次commit版本，可以这样

git reset --hard HEAD^

我们再设想一下，今天回退到了某一个版本，可是第二天后悔了该怎么办？

此时你可以使用

git reflog

可以发现git帮你记住了每一次命令，你又可以愉快地回到你想要的版本了

  • HEAD指向的版本就是当前版本，因此，Git允许我们在版本的历史之间穿梭，使用命令git reset --hard commit_id。

  • 穿梭前，用git log可以查看提交历史，以便确定要回退到哪个版本。

  • 要重返未来，用git reflog查看命令历史，以便确定要回到未来的哪个版本。

    ​ ——廖雪峰老师的官网

小结：

  1. 如果只是工作空间的撤销，只需git checkout <file>

  2. 如果add到了暂存区，可以先使用git reset HEAD <file>,回到1，然后再对工作空间的修改进行撤销

  3. 如果已经commit到了版本库，可以git reset --hard HEAD <commitId>撤销操作

现在头昏脑胀的你，不仅commit 还把错误的内容推送到了远程仓库，现在你真的完了。。

工作区 简单来说新建的文件夹就是你的工作区

版本库 git add 后添加到版本库

暂存区 git commit 添加至暂存区

远程仓库

登录github点击“new”创建一个新的仓库

填写Repository name 就是给仓库起个名字

Description 是可选的

默认是public

点击“Create repository”确定后就创建了新的git仓库

接下来你可以克隆这个远程仓库到本地

git clone git@github.com:BLinL/test.git

现在本地的的仓库就和远程仓库关联上了

远程仓库的名字默认叫origin

你可以在本地添加文件或者修改

然后推送到远程仓库

git push origin master

你的同事也可以克隆这个仓库

另一种情况是在本地已经存在一个仓库，你想将它放在github上

同样的在github上创建新的仓库，步骤不再赘述

接着把本地仓库和github上的仓库关联

github已经给出了办法

git remote add origin git@github.com:BLinL/test2.git
git push -u origin master

-u 不仅会把本地仓库master分支内容推送到远程仓库master分支，而且会把本地master分支和远程master分支关联，仅在第一次推送需要
